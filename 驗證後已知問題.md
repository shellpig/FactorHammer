## [P12-C-startup-overlay-ux] Launcher 等前後端 ready 才開頁面，導致進入畫面體感很久 — ⏳ 待實作

> 2026-05-21 使用者回報：目前啟動後會停在 launcher 視窗，顯示 `Waiting for frontend (3000) and backend (8000) to be ready (max 60s) ...`。Next.js console 也會被 health polling 刷出大量 `GET / 307` / `GET /dashboard 200`。雖然 `[P12-C-cold-start-race]` 已解決，但使用者體感變成「進入畫面都要很久」。

### 背景

`[P12-C-cold-start-race]` 的修法讓 `run_factorhammer.bat` 在開瀏覽器前同時等待:

- frontend `http://localhost:3000`
- backend `http://localhost:8000/api/health`

這能避免 backend 冷啟動時 dashboard 先打 API、閃 fetch 錯誤或跳錯 modal；但代價是瀏覽器要等 backend 完整 import pandas / pyarrow / duckdb / FastAPI 等依賴後才開。

### 問題

- 使用者在 launcher 黑窗等待，無法先看到 FactorHammer 畫面。
- health polling frontend 會反覆命中 `/`，Next.js redirect 到 `/dashboard`，console 出現大量 `GET / 307` / `GET /dashboard 200`，看起來像頁面一直被打。
- 這是 UX 問題，不是 dashboard route 本身慢；log 顯示 `/dashboard` 多數只需 40–80ms。

### 預期 UX

先進入專案畫面，但用上層 blocking overlay 擋住主要互動，顯示：

```text
工具初始化中...
```

backend ready 後才解除 overlay，接著執行既有 token onboarding 流程。

### 建議修法

1. **Launcher 改為只等 frontend ready 就開瀏覽器**
   - `run_factorhammer.bat` 開 browser 前只 poll `http://localhost:3000/dashboard` 或 `http://localhost:3000`。
   - 不再等待 `http://localhost:8000/api/health` 才開瀏覽器。
   - console 文案改成類似：`Waiting for frontend (3000) to be ready... Backend will continue initializing in app.`

2. **Dashboard 增加 startup overlay 狀態**
   - 在 `DashboardPageClient` 內新增 backend health check state，例如 `backendReady` / `backendTimedOut`。
   - 首次載入時輪詢 `/api/health`。
   - `backendReady=false` 時顯示全頁 overlay：`工具初始化中...`。
   - overlay 期間不要觸發 dashboard payload fetch。

3. **調整 token onboarding 時序**
   - 等 `/api/health` 成功後，才呼叫 `/api/config/secrets/status`。
   - 無 token：顯示既有 `TokenSetupDialog`。
   - 有 token：設定 `finmindReady=true`，解除 dashboard fetch gate。
   - `/api/health` 未 ready 時，不應誤判為 token 缺失，也不應先跳 TokenSetupDialog。

4. **backend timeout 顯示明確錯誤**
   - 若 health check 超過合理時間（建議 60 秒）仍失敗，overlay 文案改為：
     - `後端啟動逾時，請確認 FactorHammer-Backend-8000 視窗是否有錯誤。`
   - 保留重試按鈕或繼續輪詢，避免使用者只能重新整理。

5. **保留既有 cold-start race 防護**
   - `secrets/status` retry / catch 收斂邏輯不可退回舊版。
   - dashboard payload fetch 仍必須 gated on backend ready + secrets checked + finmind ready。

### 影響範圍

- `run_factorhammer.bat`
- `web/src/components/dashboard/dashboard-page-client.tsx`
- 可能新增小型元件：`StartupOverlay` 或共用 loading overlay
- 既有 `TokenSetupDialog` 行為需保留

### 驗收標準

1. 後端冷啟動慢於 frontend 時：瀏覽器先開 `/dashboard`，第一眼看到 `工具初始化中...` overlay。
2. overlay 期間不顯示 dashboard fetch error banner。
3. overlay 期間不先跳 TokenSetupDialog。
4. backend ready + `.env` 無 `FINMIND_TOKEN`：overlay 結束後跳 TokenSetupDialog。
5. backend ready + token 已設定：overlay 結束後 dashboard 正常載入。
6. backend 啟動失敗或被關閉：60 秒後顯示後端啟動逾時文案。
7. Next.js console 不再因 launcher 同時等待 frontend/backend 而大量刷 `GET /` / `GET /dashboard`。

### 建議測試

- `web/` `npx tsc --noEmit`
- `web/` `pnpm test -- --run`
- 補 dashboard client 測試：
  - health pending 時顯示 `工具初始化中...`
  - health pending 時不呼叫 dashboard payload
  - health success 後才呼叫 secrets/status
  - health timeout 顯示逾時文案
- 手動：冷啟動 `run_factorhammer.bat`，確認先看到頁面 overlay，而不是 launcher 黑窗長等。

## [P12-C-onboarding-race] Dashboard fetch 與 secrets/status 競賽，乾淨環境先閃錯誤再跳 modal — ✅ 已修正、待手動驗證

> 2026-05-21 使用者回報：從 GitHub 下載 zip、`install.bat` 完成後執行，**進入畫面時並沒有先跳 FINMIND_TOKEN 強制設定 modal，而是先進到個股分析頁面、畫面顯示 fetch 失敗錯誤，過一陣子才跳 modal**。應該一進去就跳這個 modal 才對。

### 根因

`web/src/components/dashboard/dashboard-page-client.tsx` 內兩個 `useEffect` 競賽：

1. 第一個 `useEffect` 呼叫 `apiGet("/api/config/secrets/status")` —— async HTTP，需一次 round-trip
2. 第二個 `useEffect` 從 localStorage 還原 `qt-last-symbol` 後**立刻** `setSymbol(savedSymbol)`，觸發 `useDashboard` SWR fetch

`useDashboard` 只看 `symbol !== null` 就發 `/api/dashboard/payload`，**沒有 gate FinMind 狀態**。乾淨環境（`.env` 無 token + localStorage 有上次 symbol）下：
- ② 立刻打後端 → 後端因無 FinMind token 回錯 → UI 顯示「分析失敗：...」
- 等一陣子 ① 才回來 → modal 才彈出

規格 12-C 原本沒寫明「status 還在 loading 時要 gate 住 dashboard fetch」，已於本次補進規格書 12-C「Dashboard 入口整合」與驗收表（commit 待補）。

### 修法（方法 A：Gate dashboard fetch on secrets status）

`dashboard-page-client.tsx`：

- 新增 `secretsChecked` / `finmindReady` 兩個 state
- 第一個 `useEffect` 完成後依結果設這兩 state；status 失敗時 `finmindReady = true`（規格 12-C 降級 UX）
- 第二個 `useEffect` 拆兩段：
    - 立刻還原 `market` / `pendingSymbol`（純 UI 狀態）
    - 新增第三個 `useEffect` gated on `secretsChecked && finmindReady` 才 `setSymbol(savedSymbol)`，觸發 `useDashboard`
- `<TokenSetupDialog onSaved>` 內補 `setFinmindReady(true)`，token 存好後解除 gate 讓 dashboard 開始抓

### 影響範圍

- 修正後乾淨環境一進站直接看到強制 modal，**不再閃過「分析失敗」紅色 banner**
- 已設 token 的環境：行為與舊版一致（`secrets/status` 回 `finmind: true` → 立即 `setSymbol` → 立即 fetch）
- `secrets/status` 失敗的降級 UX 保留（`finmindReady = true` 讓 dashboard 嘗試載入）

### 待補手動驗證

1. 乾淨環境（無 `.env` 或 `FINMIND_TOKEN` 空白）+ localStorage 已有 `qt-last-symbol = 2330` → 開瀏覽器：**第一眼直接是 modal，不得閃過紅色錯誤 banner**
2. 已設 token 環境：modal 不跳、dashboard 正常載入
3. modal 內輸入有效 token → 儲存 → modal 關閉 → dashboard 自動載入（不需手動 reload）
4. modal 內輸入無效 token → 紅字錯誤、modal 保持開、dashboard 仍 gate 住

### Gate Tests（待跑）

- `web/` `npx tsc --noEmit` — 0 errors
- `web/` `pnpm test -- --run`（重點：`token-setup-dialog.test.tsx`、`p11-placeholders.test.tsx`、`quote-header.test.tsx` 三個會 render `DashboardPageClient` 的測試需全綠）
