## 三審結論

Claude 修完二審後，P15 規格仍不是只剩原本 4 點。這輪改用「規格文字、sample code、現有程式、測試契約、外部官方文件」交叉比對後，仍有 8 個需要一次修完的問題。

這 8 點都屬於規格缺口或規格與現有程式不一致；若不修，implementer 很容易照規格做出不能編譯、不能驗收，或驗證邏輯錯誤的實作。

### 2. [P1] `AiToggleSection` sample 會吃不到 async config 回填

**位置**

- `開發設計方針.md:10357-10365`

**現況**

sample code：

```tsx
const { config, mutate } = useConfig();
const ai = config?.ai ?? { enabled: false, provider: "anthropic", model: DEFAULT_MODEL.anthropic };

const [enabled,  setEnabled]  = useState<boolean>(Boolean(ai.enabled));
const [provider, setProvider] = useState<string>(String(ai.provider ?? "anthropic"));
const [model,    setModel]    = useState<string>(String(ai.model ?? DEFAULT_MODEL[provider] ?? ""));
```

若 `useConfig()` 是 SWR / async hook，首次 render 時 `config` 通常是 `undefined`。`useState` 只吃第一次初始值，後續 config 載入完成後不會自動更新 local state。

**影響**

- 設定頁可能永遠顯示預設 `anthropic / disabled`，即使 `/api/config` 已有 `deepseek / enabled`。
- F5 後設定持久化驗收會失敗。
- 使用者可能按儲存時把已存在 config 覆蓋成預設值。

**建議修正**

sample code 必須補：

- `useEffect` 在 config 載入後同步 `enabled / provider / model`
- dirty guard，避免使用者正在編輯時被 SWR refresh 覆蓋
- 測試新增「初始 undefined，後續回 deepseek config，UI 會同步」case

### 3. [P2] 15-A-2 風險表仍殘留 DeepSeek `/v1/user/balance` 與錯誤 fallback

**位置**

- `量化交易系統規格書_shellpig版.md:5010-5014`
- `量化交易系統規格書_shellpig版.md:5105`
- `測試指南.md:4426-4428`

**現況**

主規格已修成：

- DeepSeek balance：`https://api.deepseek.com/user/balance`
- 無 `/v1`
- `no_quota` 只認 `402`

但風險表仍寫：

```md
DeepSeek `/v1/user/balance` 端點規格變動
fallback：401 → invalid_key、其他 4xx → 視為 ok（key 有效但其他問題）
```

這與主規格、測試指南、DeepSeek 官方文件矛盾。

**外部來源**

- DeepSeek Get User Balance 官方文件：`GET /user/balance`  
  https://api-docs.deepseek.com/api/get-user-balance/
- DeepSeek Error Codes 官方文件：`401` 是 Authentication Fails，`402` 是 Insufficient Balance，`400 / 422` 是 request/parameter error，不應被視為 key ok。  
  https://api-docs.deepseek.com/quick_start/error_codes

**影響**

- implementer 可能留下 `/v1/user/balance`，導致 validate 打錯 URL。
- 「其他 4xx 視為 ok」會把錯誤 request 或 invalid parameter 當成可寫入 key。
- `secrets/validate` 安全模型會鬆掉，與「有填就驗證」不一致。

**建議修正**

把 `量化交易系統規格書_shellpig版.md:5105` 改成：

- 端點：`GET https://api.deepseek.com/user/balance`
- fallback：
  - `200` → `ok`
  - `401` → `invalid_key`
  - `402` → `no_quota`，可寫入
  - `429 / 5xx / network timeout` → `unreachable`
  - 其他 `4xx` → `invalid_key` 或 `unreachable`，不得視為 ok，不得寫入

### 4. [P2] FinMind 驗證端點仍寫 `/v4/login`，與現有實作不一致

**位置**

- `量化交易系統規格書_shellpig版.md:5014`
- 現有程式：`src/services/config_service.py:178-213`

**現況**

P15-A-2 表格寫：

```md
FinMind 沿用既有 `/v4/login`
```

但現有 `validate_finmind_token()` 實作是：

```py
url = "https://api.finmindtrade.com/api/v4/data"
params = {
    "dataset": "TaiwanStockInfo",
    "data_id": "2330",
    ...
    "token": token,
}
```

也就是用 FinMind data endpoint 驗證 token，而不是 `/v4/login`。

**外部來源**

- FinMind 官方教學使用 `https://api.finmindtrade.com/api/v4/data` 並以 `dataset = "TaiwanStockInfo"` 取得台股資訊。  
  https://finmind.github.io/tutor/TaiwanMarket/Technical/
- FinMind Swagger 文件列出 `/api/v4/data` 與 `TaiwanStockInfo`。  
  https://api.finmindtrade.com/docs

**影響**

- implementer 可能誤改已存在且已測過的 FinMind 驗證路徑。
- 會造成 P12-B 既有測試與 P15-A-2 新測試方向不一致。
- 使用者已明確要求 15-A secrets/validate：FinMind 必填；其他 provider 選填但有填就驗證。這裡不應順手換 FinMind 驗證方式。

**建議修正**

把 `FinMind 沿用既有 /v4/login` 改成：

```md
FinMind 沿用既有 `validate_finmind_token()`，即透過 `https://api.finmindtrade.com/api/v4/data` 查 `TaiwanStockInfo` 輕量資料驗證 token；P15-A-2 只包裝輸出為 `ValidationResult`，不改驗證 endpoint。
```

### 5. [P1] `/api/ai/chat` sample 使用 `get_advisor()`，但現有 dependency 不存在

**位置**

- `開發設計方針.md:10965-10968`
- 現有程式：`api/deps.py:1-10`

**現況**

15-B sample：

```py
@router.post("/chat")
async def post_ai_chat(request: ChatRequest):
    advisor = get_advisor()
```

但目前 `api/deps.py` 只有：

```py
def get_manager() -> JobManager:
    return get_job_manager()
```

沒有 `get_advisor()`。

**影響**

- implementer 若照 sample 寫，會出現 import/name error。
- FastAPI dependency pattern 不一致；其他 routers 使用 `Depends(get_manager)`。
- 測試 mock advisor 時沒有明確 injection seam，15-B / 15-D SSE 測試會難寫。

**建議修正**

規格二選一，必須明確：

1. 在 `api/deps.py` 新增：

```py
from src.ai.advisor import AIAdvisor

def get_advisor() -> AIAdvisor:
    return AIAdvisor()
```

並在 router 用：

```py
async def post_ai_chat(
    request: ChatRequest,
    advisor: AIAdvisor = Depends(get_advisor),
):
```

2. 或明確寫成 router 內直接 `advisor = AIAdvisor()`，並補測試 monkeypatch 位置。

建議採第 1 種，較符合現有 `api/deps.py` pattern。

### 7. [P2] provider adapter `complete()` 介面與 DeepSeek/OpenAI 額外 kwargs 仍可能漂移

**位置**

- `量化交易系統規格書_shellpig版.md:4940`
- `量化交易系統規格書_shellpig版.md:4993`
- 現有程式：`src/ai/advisor.py:148-166`
- 現有程式：`src/ai/advisor.py:302-314`
- 現有測試 stub：`tests/test_advisor.py:95-112`、`tests/test_advisor.py:313-320`

**現況**

P15-A 要 dashboard 在 provider 為 OpenAI / DeepSeek 時傳：

```py
response_format={"type": "json_object"}
```

DeepSeek 額外傳：

```py
thinking={"type": "disabled"}
```

但現有 `BaseProviderAdapter.complete()` abstract signature 與多個 test stub 都只有：

```py
def complete(*, model, system_prompt, messages, tools) -> dict
```

**外部來源**

- DeepSeek Create Chat Completion 官方文件列出 `thinking`，`type` 可為 `enabled / disabled`，預設 `enabled`。  
  https://api-docs.deepseek.com/api/create-chat-completion
- 同一份文件列出 `response_format`，`{"type": "json_object"}` 可啟用 JSON Output。  
  https://api-docs.deepseek.com/api/create-chat-completion

**影響**

- 若 `generate_stock_dashboard_analysis()` 對 injected adapter / test stub 傳 `response_format` 或 `thinking`，測試會噴：

```text
TypeError: complete() got an unexpected keyword argument 'thinking'
```

- 規格目前容易讓 implementer 只改真實 OpenAIAdapter，忘記 base interface 與測試 double。

**建議修正**

規格明確補一條：

- `BaseProviderAdapter.complete()` signature 加：

```py
response_format: dict[str, Any] | None = None,
thinking: dict[str, Any] | None = None,
```

- Anthropic / Gemini adapter 可接受但忽略，或 base 規格要求非 OpenAI provider 不傳。
- 所有測試 stub `complete()` 加 `**kwargs` 或同名 optional kwargs。
- 新增測試：
  - OpenAI dashboard 收到 `response_format`
  - DeepSeek dashboard 收到 `response_format` + `thinking`
  - Anthropic / Gemini 不收到 DeepSeek-only kwargs

### 8. [P2] `useAIChat` 回傳契約與 sample code 不一致

**位置**

- `量化交易系統規格書_shellpig版.md:5197`
- `開發設計方針.md:11029-11132`
- `開發設計方針.md:11137-11138`

**現況**

規格表寫：

```ts
useAIChat() 回 { messages, send, abort, streaming, error }
```

但 sample code 最後回傳：

```ts
return { messages, send, abort, streaming };
```

而 chat-page-client sample 也只解構：

```tsx
const { messages, send, abort, streaming } = useAIChat();
```

錯誤目前是塞進最後一則 assistant message 的 `error` 欄位，不是 top-level `error` state。

**影響**

- implementer 不知道要不要做 top-level `error`。
- 測試若照規格表 mock `{ error }`，component 若照 sample 不使用，會出現契約漂移。
- error bubble 驗收可以過，但 hook public API 與規格表不一致。

**建議修正**

二選一，規格要統一：

1. 移除 top-level `error`，把 required file 表格改成：

```ts
useAIChat() 回 { messages, send, abort, streaming }
```

並明確寫「error event 寫入 assistant message 的 `error` 欄位」。

2. 或 sample code 補：

```ts
const [error, setError] = useState<string | null>(null);
...
return { messages, send, abort, streaming, error };
```

建議採第 1 種，因現有 sample 與 UI bubble 設計已走 message-level error。
